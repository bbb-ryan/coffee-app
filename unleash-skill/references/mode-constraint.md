# Constraint Embrace

Treat limitations as generative fuel. The constraint isn't blocking the solution — it IS the solution.

## Cognitive Move

Stop fighting constraints. Ask what they make POSSIBLE. A budget limit forces you to find what's essential. A deadline reveals which scope was always unnecessary. The constraint does the thinking for you.

## Process

1. **Name every constraint specifically.** Not "limited budget" — "$5K and 3 weeks." Not "small team" — "1 designer, 2 engineers."
2. **Categorize:** Hard (physics, legal, committed budget), Soft (changeable at cost), Assumed (feels real but is a choice).
3. **For each hard constraint: "What does this ENABLE?"** Every constraint closes doors, opens others. Tiny budget → forced organic distribution → healthier business.
4. **Find the gift.** One constraint, fully embraced, usually reshapes the solution better than unlimited resources would.
5. **Design FROM the constraint.** Don't mitigate. Don't work around. Make it the organizing principle.

## Quality Gate

PASS: ≥1 constraint reframed from obstacle to advantage, constrained solution is genuinely different (not compromised unconstrained version), can articulate why constrained is BETTER in some dimension.
FAIL: Constraints acknowledged but worked around, output is cope ("we can still do a lot!"), no specific constraint identified as generative.

If output quality is unclear, read `./examples.md` (constraint section) for calibration.

Chains: subtraction, first-principles. Tension: scale-break (productive — "which to break, which to embrace?").
