# Pattern Bridge

Find a structural pattern in one domain and transfer it to solve a problem in another. The insight is the shape that connects them.

## Cognitive Move

Look past surface content. Find the structural shape of the problem. Search other domains for the same shape with a known solution. Bateson's "pattern which connects." Munger's mental model lattice.

## Process

1. **Abstract to structure.** Strip domain language. "Users sign up but don't activate" → commitment-action gap. "Codebase hard to change" → accumulated coupling. Name the pattern.
2. **Search for that pattern elsewhere.** Check: biology (evolution, ecosystems, immune systems), physics/engineering, urban planning, economics, music/art, military strategy, psychology. Also: user's own domains (read mycelium profile if available).
3. **Find structural match.** Must be structural, not metaphorical. "Like a garden" = no. "Same growth/pruning dynamic, solved by X" = yes.
4. **Transfer the mechanism.** What does the source domain do about this pattern? How to adapt it?
5. **Validate.** Does mechanism work in new context? Where does analogy break?

User can specify: `/unleash bridge music architecture` or leave open.

## Quality Gate

PASS: Pattern named specifically, source mechanism articulated, transfer produced actionable approach, analogy limits identified.
FAIL: Metaphorical bridge ("our product is like a garden"), pattern forced, source domain mentioned but mechanism not extracted.

If output quality is unclear, read `./examples.md` (pattern-bridge section) for calibration.

Chains: first-principles (find abstract pattern), empathy (bridge from user's own expertise).
