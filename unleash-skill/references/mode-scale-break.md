# Scale Break (10x)

Reject incremental improvement. Demand 10x — it forces you off the current curve entirely. 10% traps you in optimizing the wrong thing.

## Cognitive Move

Treat the current solution as version 0.1 of a fundamentally wrong paradigm. The question isn't "how do we make this better?" — it's "what would make this unnecessary?" Look for the approach that makes the current one look like horse-and-buggy optimization.

## Process

1. **Name the current approach and its ceiling.** Every approach has a maximum. What's the asymptote?
2. **Define 10x literally.** What would 10x output/reach/speed/simplicity/impact mean in measurable terms?
3. **Identify what breaks at 10x.** Architecture, team, cost model, distribution — something can't survive. That breaking point reveals the real constraint.
4. **Find the paradigm shift.** What category change makes 10x trivial? Usually: manual→automated, sync→async, centralized→distributed, product→platform.
5. **Pressure-test.** Technically possible? What's the MVP? First step toward 10x, not toward 10%?

## Quality Gate

PASS: Identified ceiling on current approach, 10x revealed a breaking point invisible at normal scale, paradigm-level alternative emerged with rough feasibility path.
FAIL: "10x" used as motivational amplifier, same plan with bigger numbers, no breaking point identified.

If output quality is unclear, read `./examples.md` (scale-break section) for calibration.

Chains: first-principles, reframe. Tension: subtraction (productive — "most stripped-down version of 10x paradigm?").
